# 记忆结构与生命周期说明

本文说明当前协作记忆系统如何存储、提取、召回、更新和管理记忆，并补充说明后续可以落地的注意力加权模型。

## 1. 记忆的存储结构

系统把记忆拆成四层结构：记忆实体、事件日志、状态快照、会话轨迹。

### 1.1 记忆实体

每条记忆是一个 `MemoryItem`，持久化为一份 Markdown 文件，文件名格式类似：

```text
mem_xxxxxxxx-<scope>-<type>.md
```

实体字段包括：

```text
id             记忆唯一 ID
type           记忆类型
content        给人看的自然语言内容
scope          适用场景/领域
subject        事实主体
target         作用对象
object         关键对象
predicate      关系/动作
source         来源
confidence     置信度
status         生命周期状态
evidence       原始证据
applies_when   可应用场景
tags           检索标签
validity       有效期/任务期信息
created_at     创建时间
updated_at     更新时间
supersedes     被它替代的旧记忆
```

当前主要 `type` 有：

```text
preference    偏好，例如“老公的礼物偏好/背景：咖啡”
constraint    约束，例如“预算在 1000 元以内”
workflow      做事方式，例如“周报保留风险和负责人”
candidate     候选方案，例如“曾推荐智能阳台养护套装”
decision      当前任务已选方案，例如“本次给老公选定信乐烧莲花盆”
history       已发生事实，例如“已经给老公买过胡桃木掌托”
context_fact  背景事实，例如“父亲膝盖不好”
```

`scope` 用来隔离场景，目前包括：

```text
relationship_gift   送礼/关系场景
life_family_travel  家庭旅行
work_report         工作汇报/项目材料
study_plan          学习计划
research_review     文献综述/研究设计
general             无法归类的通用记忆
```

### 1.2 事件日志

所有记忆变化会追加到 `_events.jsonl`，每行是一次事件：

```text
action       add / update / downgrade / archive / delete / reset
memory_id    被操作的记忆 ID
detail       事件说明
version      版本号，例如 M6
timestamp    发生时间
```

事件日志用于追溯“为什么版本发生变化”，也是 eval trace 判断记忆是否真实写入的依据。

### 1.3 状态快照

`_state.json` 存当前版本号：

```text
version
updated_at
```

运行时还会生成 snapshot，按状态分组：

```text
active
superseded
archived
deleted
```

只有 `active` 记忆会参与正常召回。`superseded`、`archived`、`deleted` 会保留在历史里，但默认不参与任务回答。

### 1.4 会话轨迹

eval harness 会把每轮对话、tool call、memory actions、snapshot 写进 trace。它不等同于长期记忆库，但用于验证：

```text
用户说了什么
系统调用了什么工具
新增/删除/降级了哪些记忆
当时 active snapshot 是什么
回复是否和真实 memory_actions 对齐
```

## 2. 记忆的提取与召回

### 2.1 如何识别需要提取的记忆

每轮消息先经过命令识别。如果是管理命令，例如：

```text
reset / 清空 / 重置 / 清除当前记忆
show / 展示 / 查看
delete / 删除某条记忆
downgrade / 降权
archive / 归档
```

系统直接执行管理动作，不进入普通记忆提取。

普通对话中，系统通过以下信号判断是否需要提取记忆：

```text
显式保存信号：以后、请记住、记住、加入这条记忆
偏好信号：喜欢、不喜欢、讨厌、偏好
约束信号：不要、不能、不用、只用于、仅用于
任务方式信号：先、最后、风险、负责人、下一步、可复现
送礼历史信号：送过、已经送、买过、买了、下单
决策信号：选定、决定、就这个、满意
领域信号：程序员、养花、养金鱼、旅行、周报、复习、文献综述等
```

如果用户说：

```text
加入这条记忆“已经买过胡桃木掌托和信乐烧莲花碗”
```

系统会先剥掉“加入这条记忆”外壳，只用内部 payload 做类型判断：

```text
已经买过胡桃木掌托和信乐烧莲花碗
```

然后因为命中 `买过`，优先判为 `history`，而不是 `preference`。

### 2.2 如何确定 scope 和 type

系统先推断 `scope`，再推断 `type`。

`scope` 主要由当前文本和上下文决定。例如：

```text
给老公买礼物 -> relationship_gift
家庭亲子旅行 -> life_family_travel
老板周报 / 项目风险 -> work_report
高数复习 / 番茄钟 -> study_plan
RAG 文献综述 -> research_review
```

送礼场景会进一步识别收礼人：

```text
女朋友 -> girlfriend
老公/丈夫/先生 -> husband
老婆/妻子 -> wife
妈妈/母亲 -> mother
爸爸/父亲 -> father
朋友 -> friend
同事 -> colleague
```

购买或送出事实会优先变成 `history`：

```text
已经买过 X
已经买了 X
下单了 X
以前送过 X
已经送过 X
```

当前任务选定但未发生的事实会变成 `decision`：

```text
本次给老公选定 X
就这个
定这个
```

偏好类信息才会进入 `preference`：

```text
他喜欢咖啡
她喜欢玫瑰金
孩子喜欢自然和动物
```

### 2.3 如何召回记忆

召回流程如下：

1. 根据当前用户输入推断当前 `scope`。
2. 从输入中抽取关键词，例如预算、礼物对象、风险、番茄钟、文献综述等。
3. 遍历所有 `active` 记忆。
4. 跳过明显污染的记忆。
5. 满足以下条件之一就召回：
   - 当前 `scope` 命中记忆的 `scope` 或 `applies_when`
   - 当前关键词命中记忆的 `content/scope/subject/target/object/predicate/tags`

召回后，系统会把相关记忆交给具体领域的回答函数使用。例如送礼场景会组织成礼物画像：

```text
预算约束
收礼人偏好/背景
以前送过/买过的历史
当前已选方案
曾推荐候选
禁忌和重复过滤
```

在送礼画像里，还会按收礼人过滤，避免“老公喜欢养金鱼”污染“给妈妈买礼物”。

## 3. 记忆的更新与管理

### 3.1 对话中如何更新记忆

每轮消息的处理顺序是：

```text
1. 尝试识别管理命令
2. 检查是否和旧记忆冲突
3. 提取新记忆候选
4. 去重
5. 写入 MemoryStore
6. 检索相关 active 记忆
7. 生成回复
8. 对部分场景记录 assistant candidate
```

新增记忆时：

```text
status = active
version + 1
写入 mem_xxx-<scope>-<type>.md
追加 _events.jsonl
更新 _state.json
```

删除记忆时：

```text
status = deleted
version + 1
保留文件和事件
默认不再召回
```

重置记忆时：

```text
清空 items
清空 events
version 回到 M0
删除当前 storage_dir 下的 memory md 文件
写入 reset event 和 _state.json
```

### 3.2 存量记忆改变或降级时如何处理

当前实现有几类状态变化：

```text
update      直接修改字段，保留 active 状态
downgrade   标记为 superseded，并把 confidence 降到最多 0.45
archive     标记为 archived
delete      标记为 deleted
reset       清空当前记忆库
```

降级通常发生在用户反馈推翻或缩小旧规则时，例如：

```text
少步行不适用
不要硬套紫色
不再推荐某个礼物
```

降级后的记忆不会参与正常 `active()` 召回，但仍会出现在 snapshot 的 `superseded` 分组，方便追溯。

删除和归档也类似：数据保留，默认不参与回答。

### 3.3 注意力权重和加权评分

当前实现的召回更接近“规则过滤 + 命中即用”，还没有完整的数值排序器。实际注意力权重由以下隐式规则决定：

```text
status        只有 active 默认可用
scope         当前场景命中会被召回
term_hit      关键词命中会被召回
pollution     明显污染记忆会被跳过
confidence    downgrade 会降低 confidence，但当前召回未完整按 confidence 排序
recency       有 created_at/updated_at，但当前召回未按时间排序
validity      有 current_task/past/long_term，但当前召回只在部分领域逻辑中使用
```

为了更稳定地决定哪条记忆注意力更高，建议升级为显式打分。一个可落地的公式是：

```text
score =
  0.30 * scope_score
+ 0.20 * semantic_match_score
+ 0.15 * type_priority_score
+ 0.10 * status_score
+ 0.10 * validity_score
+ 0.05 * confidence_score
+ 0.05 * recency_score
+ 0.05 * evidence_quality_score
- penalty_score
```

各项含义：

```text
scope_score
  当前 scope 完全一致最高；applies_when 命中次之；跨域最低。

semantic_match_score
  用户文本和 memory 的 content/object/tags/subject/target 匹配越多越高。

type_priority_score
  不同任务阶段优先级不同。
  做推荐时：constraint/history/decision > preference > candidate。
  做风格化输出时：workflow > preference > context_fact。
  做复盘时：history/decision > candidate。

status_score
  active = 1；superseded/deleted/archived = 0，除非用户明确查看历史。

validity_score
  current_task 在当前任务内更高；past 适合避重；long_term 适合稳定偏好。

confidence_score
  使用 MemoryItem.confidence。

recency_score
  最近更新的 current_task 记忆略高，但不能覆盖明确 long_term 约束。

evidence_quality_score
  用户明确说“请记住/已经买过/以前送过”高于模型推断。

penalty_score
  主体不一致、收礼人不一致、被用户否定、疑问句误抽取、过期上下文等都扣分。
```

送礼场景的注意力优先级可以单独定义：

```text
1. 当前用户明确输入
2. 当前任务 decision
3. history：已买/已送/已选过，主要用于避重
4. constraint：预算、禁忌、不要重复
5. recipient preference/context：收礼人兴趣、职业、偏好
6. candidate：曾推荐过，仅用于避免重复或解释
7. general 记忆：只有没有更具体 scope 时使用
```

这样可以避免几个常见错误：

```text
把“买过”误判为 preference
把 A 收礼人的偏好套到 B 收礼人身上
把旧候选当成最终已买历史
让 general 记忆压过 relationship_gift/work_report 等具体 scope
让已删除或已降级记忆继续影响回答
```

## 4. 一个完整例子

用户说：

```text
加入这条记忆“已经买过胡桃木掌托和信乐烧莲花碗”
```

如果当前会话或 active 记忆能判断正在给老公选礼物，应该写成：

```text
type: history
scope: relationship_gift
subject: user
target: husband
predicate: gave
content: 已经给老公买过胡桃木掌托、信乐烧莲花碗
validity: {"time_scope": "past"}
```

后续再让系统继续给老公选礼物时，这条记忆会作为 `history` 被召回，作用是：

```text
不要重复推荐胡桃木掌托
不要重复推荐信乐烧莲花碗
核算剩余预算时把它们当成已发生购买
需要补充礼物时选择不同品类
```

如果完全没有任何领域上下文，至少应该写成：

```text
type: history
scope: general
content: 已经买过胡桃木掌托和信乐烧莲花碗
```

也就是说，购买事实不应该落到 `preference`。
